# 上线检查清单

## 一、部署前（Pre-Deploy）

### 1. 代码与配置
- [ ] 已合并主分支并通过 CI（Django test + pytest）
- [ ] `.env` 已配置生产参数（关闭 DEBUG，正确 ALLOWED_HOSTS）
- [ ] `DJANGO_SECRET_KEY`、`FERNET_KEY`、`MYSQL_*`、`REDIS_URL` 已配置
- [ ] `OPS_ADMIN_USERNAMES` 已按运维名单配置

### 2. 基础设施
- [ ] MySQL 主库/从库可连通，账号权限最小化
- [ ] Redis 可连通，容量与淘汰策略符合预期
- [ ] 应用服务器时区与 NTP 时间同步

### 3. 应用自检
- [ ] 执行 `python manage.py preflight_check` 成功
- [ ] 执行 `python manage.py migrate --plan` 检查迁移变更
- [ ] 执行 `python manage.py list_ops_admin` 确认运维权限名单

## 二、部署中（Deploy）

### 1. 发布步骤
- [ ] 备份数据库（至少结构 + 关键业务表）
- [ ] 执行数据库迁移：`python manage.py migrate`
- [ ] 启动应用：`python manage.py runserver`（生产建议 Gunicorn/Uvicorn）
- [ ] 启动 Celery Worker：`celery -A myproject worker -l info`
- [ ] 启动 Celery Beat：`celery -A myproject beat -l info`

### 2. 灰度验证
- [ ] 打开 `/api/health/` 检查 DB/Cache 状态
- [ ] 打开 `/swagger/` 校验 API 文档可访问
- [ ] 用测试账号跑通核心接口：授权、采集、库存同步

## 三、部署后（Post-Deploy）

### 1. 功能验证
- [ ] `GET /api/health/` 返回 `status=ok`
- [ ] `GET /metrics` 可被监控系统采集
- [ ] 死信队列接口可用：`/api/ops/dead-letter/`
- [ ] 运维权限自检：`/api/ops/whoami/`

### 2. 稳定性观察（首 30~60 分钟）
- [ ] API 5xx 比例 < 1%
- [ ] Celery 任务失败率可控，无明显积压
- [ ] MySQL 慢查询未显著上升
- [ ] Redis 内存与连接数在安全阈值内

### 3. 回滚预案
- [ ] 准备上个稳定版本镜像/代码包
- [ ] 可快速停止新 Worker 并切回旧版本
- [ ] 如需回滚 DB，遵循预先验证过的回滚脚本

## 四、常用命令速查

```bash
# 依赖安装
pip install -r requirements.txt

# 迁移
python manage.py migrate

# 发布前自检
python manage.py preflight_check

# 运行测试（SQLite 测试配置）
python manage.py test --settings=myproject.settings_test
pytest

# 运维权限
python manage.py list_ops_admin
python manage.py sync_ops_admin_from_settings --dry-run
python manage.py prune_ops_admin --dry-run --keep-superuser
```

## 五、短信登录与风控验收（新增）

### 1. 必要环境变量
- [ ] `SMS_PROVIDER_CHAIN` 已配置（建议 `aliyun,tencent,mock`）
- [ ] `SMS_GLOBAL_HOURLY_LIMIT` 已配置（默认 `10000`）
- [ ] `DEVICE_PHONE_DAILY_LIMIT` 已配置（默认 `5`）
- [ ] `SMS_CODE_TTL_SECONDS`、`SMS_SEND_MIN_INTERVAL_SECONDS` 已按业务设置

### 2. 主流程验收（curl）

```bash
# 1) 发送验证码（开发环境 DEBUG=True 时返回 code）
curl -X POST "http://127.0.0.1:8000/api/auth/sms/send-code" \
  -H "Content-Type: application/json" \
  -H "X-Device-ID: qa-device-001" \
  -d '{"phone":"13800138000","country_code":"86"}'

# 2) 验证码登录/注册（将 <CODE> 替换为上一步返回）
curl -X POST "http://127.0.0.1:8000/api/auth/mobile/login" \
  -H "Content-Type: application/json" \
  -H "X-Device-ID: qa-device-001" \
  -d '{"mobile":"13800138000","country_code":"86","code":"<CODE>","agreed_privacy":true}'

# 3) 再发验证码用于注销
curl -X POST "http://127.0.0.1:8000/api/auth/sms/send-code" \
  -H "Content-Type: application/json" \
  -H "X-Device-ID: qa-device-001" \
  -d '{"phone":"13800138000","country_code":"86"}'

# 4) 账号注销（将 <ACCESS_TOKEN> 与 <CODE2> 替换为真实值）
curl -X DELETE "http://127.0.0.1:8000/api/user/account/" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -d '{"code":"<CODE2>","reason":"qa-acceptance"}'
```

### 3. 预期结果
- [ ] 登录接口当 `agreed_privacy=false` 返回 400 且提示“必须同意隐私协议”
- [ ] 登录成功返回 `access/refresh`，且 `user.mobile` 已脱敏（如 `138****8000`）
- [ ] 注销后用户 `is_active=false`，并写入 `AccountDeletionLog`
- [ ] 验证码只能成功使用一次（第二次校验应失败）

### 4. 风控路径验收
- [ ] 同设备 24 小时关联手机号超过阈值后，再登录返回 403（设备黑名单）
- [ ] 短信发送超过全站小时阈值后，`/api/auth/sms/send-code` 返回 429
- [ ] 短时间重复发送同手机号验证码，触发 429（冷却限制）

### 5. 一键脚本验收
- [ ] 可执行脚本：`python scripts/verify_mobile_auth_flow.py --base-url http://127.0.0.1:8000 --mobile 13800138000`
- [ ] 脚本四步均成功后，视为主链路验收通过
