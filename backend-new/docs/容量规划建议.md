# 容量规划建议

## 1. 目标
- 为千万级 SKU 场景提供可执行的容量估算方法。
- 明确 MySQL、Redis、Celery 的扩容触发点与阶段策略。

---

## 2. 输入参数（建议先收集）
- 日均活跃商家数 `M`
- 单商家平均 SKU 数 `S`
- 总 SKU 规模 `T = M * S`
- 峰值 API QPS `Q_api`
- 峰值采集任务提交速率（条/秒）`Q_collect`
- 峰值库存同步更新速率（条/秒）`Q_sync`
- 平均商品记录大小（KB）`R_product_kb`
- 平均缓存对象大小（KB）`R_cache_kb`

---

## 3. MySQL 容量估算

## 3.1 存储估算
- 主表 `Product` 体量近似：
  - `Size_product ≈ T * (R_product_kb + 索引开销系数)`
- 索引开销系数经验值：
  - B-Tree 索引场景可按数据体积的 `30%~80%` 估算
- 示例：
  - `T=10,000,000`，单行 1.2KB，索引系数 50%
  - 约 `10,000,000 * 1.2KB * 1.5 = 18,000,000KB ≈ 17.2GB`

## 3.2 写入吞吐估算
- 峰值写入 IOPS 近似：
  - `IOPS_write ≈ Q_collect * 每条写入页数 + Q_sync * 每条更新页数`
- 监控阈值建议：
  - CPU > 70% 持续 10 分钟：预警
  - CPU > 85% 持续 5 分钟：扩容
  - 活跃连接 > max_connections * 0.8：预警

## 3.3 扩容触发点
- 单库数据量 > 300GB（或单表 > 100GB）建议进入分库分表阶段
- 慢查询每分钟 > 基线 3 倍且持续 30 分钟
- 主从延迟 > 3 秒持续 10 分钟

---

## 4. Redis 容量估算

## 4.1 内存估算
- 缓存 Key 数近似：
  - `N_cache ≈ 热点 SKU 数 + token 数 + 幂等键数 + 限流键数`
- 内存近似：
  - `Mem ≈ N_cache * (R_cache_kb + key_overhead_kb)`
- 建议预留：
  - 至少 `30%` 以上空闲内存，防止高峰抖动和淘汰风暴

## 4.2 扩容触发点
- 内存使用率 > 80%（预警），> 90%（扩容）
- 连接数 > 最大连接数 80%
- evictions 持续增长（缓存策略需要调整或扩容）

---

## 5. Celery/队列容量估算

## 5.1 Worker 并发估算
- 单 worker 吞吐近似：
  - `TPS_worker ≈ 并发数 / 平均任务耗时(秒)`
- 目标 worker 数：
  - `Workers ≈ 峰值任务速率 / TPS_worker * 安全系数(1.2~1.5)`

## 5.2 队列积压阈值
- 队列长度 > 5000 且 10 分钟无下降：扩容 worker
- 任务失败率 > 3%：优先排查下游依赖与重试策略

---

## 6. 分阶段容量策略

## 阶段 A（< 100 万 SKU）
- 单 MySQL 主从 + 单 Redis + 少量 worker
- 重点：索引与慢查询治理

## 阶段 B（100 万 ~ 1000 万 SKU）
- MySQL 增加只读副本，读写分离稳定运行
- Redis 主从或哨兵
- Celery 按队列拆分（采集/同步分离）

## 阶段 C（> 1000 万 SKU）
- 分库分表（按 `platform + hash(platform_product_id)`）
- 引入分片治理（如 ShardingSphere）
- 热点商家独立分片

## 8. 每周容量例行检查
- API：QPS、P95、5xx
- MySQL：连接数、慢查询、主从延迟、磁盘增长率
- Redis：内存、连接数、evictions
- Celery：队列积压、成功率、重试率、dead-letter 增速
