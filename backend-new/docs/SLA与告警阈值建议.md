# SLA 与告警阈值建议

## 1. 服务等级目标（SLA/SLO）

## API 层
- 可用性（Availability）：`>= 99.9%`
- 错误率（5xx）：`< 0.5%`（5 分钟窗口）
- 延迟（Latency）：
  - P95 `< 300ms`
  - P99 `< 800ms`

## 异步任务层（Celery）
- 任务成功率：`>= 99%`
- 队列积压恢复时间：`< 10 分钟`
- 重试后最终失败率：`< 1%`

## 数据层
- MySQL 可用性：`>= 99.95%`
- Redis 可用性：`>= 99.95%`

---

## 2. 告警分级（建议）

### P1（立即处理）
- API 5xx 比例 `> 5%` 持续 5 分钟
- `/api/health/` 返回 `503` 持续 2 分钟
- MySQL/Redis 不可用
- Celery 队列积压激增且持续上升

### P2（30 分钟内处理）
- API P95 延迟 `> 800ms` 持续 10 分钟
- Celery 失败率 `> 3%` 持续 10 分钟
- dead-letter 新增速率明显高于基线

### P3（工作时间处理）
- API P95 延迟 `> 400ms` 持续 30 分钟
- 慢查询数量超过基线 2 倍
- 缓存命中率低于目标值

---

## 3. 指标与阈值（可直接落监控）

## 3.1 API 指标
- `api_request_total`（按 path/method/status_code 聚合）
  - 告警：5xx rate > 1%（P2），>5%（P1）
- `api_request_latency_seconds`（Histogram）
  - 告警：P95 > 300ms（P3），>800ms（P2）

## 3.2 健康检查
- `/api/health/` 状态码
  - 告警：连续 3 次非 200（P1）

## 3.3 Celery 指标（建议接入）
- 队列长度（Redis list length）
  - 告警：> 5000 且 10 分钟不下降（P1）
- 任务失败率
  - 告警：> 3%（P2），> 10%（P1）
- dead-letter 增速
  - 告警：每分钟 > 基线 3 倍（P2）

## 3.4 MySQL 指标
- 活跃连接数 / 最大连接比
  - 告警：> 80%（P2），> 90%（P1）
- 慢查询数（每分钟）
  - 告警：> 基线 2 倍（P3），> 5 倍（P2）
- 主从延迟（若有）
  - 告警：> 3s（P2），> 10s（P1）

## 3.5 Redis 指标
- 内存使用率
  - 告警：> 80%（P2），> 90%（P1）
- 连接数 / 最大连接数
  - 告警：> 80%（P2）
- key 驱逐（evictions）
  - 告警：持续增长（P2）

---

## 4. 处理流程（On-Call）
1. 告警触发后 5 分钟内确认影响范围（接口、平台、功能）
2. 执行应急手册中的“临时止血”
3. 10 分钟内给出状态播报（影响、绕行、预计恢复）
4. 恢复后输出复盘（根因、修复、预防项）

---

## 5. 当前项目建议启用的最小告警集
- API 5xx 比例
- API P95
- `/api/health/` 非 200
- Celery 队列长度
- dead-letter 新增数量
- MySQL 慢查询数
- Redis 内存使用率
