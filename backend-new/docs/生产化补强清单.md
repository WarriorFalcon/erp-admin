# 生产化补强清单

## 1. 已落地能力
- 限流：按 `IP + Path + 写操作` 维度做每分钟限流（中间件）。
- 幂等：支持 `X-Idempotency-Key`，对重复写请求返回首次响应（中间件）。
- 请求追踪：支持 `X-Request-ID`，自动回传到响应头。
- 异步任务可观测：采集/库存同步/token 刷新增加关键日志。
- dead-letter 运维接口：
  - `GET /api/ops/dead-letter/?status=pending&task_name=...&page=1&page_size=20`
  - `POST /api/ops/dead-letter/{id}/replay/`
  - `GET /api/ops/replay-audit/?dead_letter_id=...`
  - `GET /api/ops/whoami/`（查看当前账号是否具备运维权限）
  - 权限：仅运维管理员可访问（JWT 登录后，用户名在 `OPS_ADMIN_USERNAMES` 或 superuser）
  - 额外授权方式：用户加入 Django 组 `ops_admin`

## 2. 推荐压测命令
- 单接口压测（Windows 安装 wrk 或使用 k6）：
  - `k6 run scripts/k6_sku_detail.js`
  - 目标：缓存命中情况下 P95 < 50ms

## 3. SQL 优化检查项
- 查询必须命中索引：`platform`、`platform_product_id`、`updated_at`。
- 禁止大偏移分页：使用游标分页或主键翻页。
- 批量写入优先：`bulk_create` / 批次 upsert。
- 高频报表查询走只读库。

## 4. 失败补偿建议
- Celery 失败任务写入 dead-letter 表（建议新增 `TaskDeadLetter`）。
- 后台提供补偿接口按任务 ID 重放。
- 外部平台错误码分级：4xx 不重试，5xx 指数退避重试。

## 5. 监控埋点建议
- API：QPS、RT(P95/P99)、4xx/5xx 比例。
- Worker：任务成功率、重试次数、积压队列长度。
- DB：慢 SQL、连接池使用率、主从延迟。
