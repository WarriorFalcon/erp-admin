# 数据库设计

## 1. 核心实体
- `PlatformToken`：平台授权 Token，密文存储 `access_token/refresh_token`，含 `expires_at`。
- `Product`：SKU 主表，存储平台商品主信息。
- `ProductVariant`：规格明细表，和 `Product` 一对多。
- `CollectionTask`：采集任务表，记录任务状态与目标 ID。
- `SyncRule`：库存同步规则（平台 + 仓库维度）。
- `InventorySyncLog`：库存同步执行日志。

## 2. ER 文字描述
- 一个 `Product` 可有多个 `ProductVariant`（`ProductVariant.product_id -> Product.id`）。
- 一个 `SyncRule` 对应多条 `InventorySyncLog`（逻辑关联：platform + warehouse_id）。
- 一个 `PlatformToken` 对应一个平台账号（`platform + account_id` 唯一）。
- 一个 `CollectionTask` 对应一次平台采集批次（target_ids JSON）。

## 3. 索引设计
- `Product.platform_product_id`：`db_index=True`
- `Product.platform`：`db_index=True`
- `Product.updated_at`：`db_index=True`
- 组合索引：
  - `Product(platform, platform_product_id)`
  - `Product(platform, updated_at)`
- 其他高频索引：
  - `CollectionTask(status, updated_at)`（可按需扩展）
  - `SyncRule(sync_enabled, platform)`
  - `PlatformToken(platform, account_id, expires_at)`

## 4. 千万级 SKU 分库分表示意
- 当前代码保留单库模型，生产建议：
  1. 逻辑分片键：`hash(platform + platform_product_id)`。
  2. 路由策略：按平台先分库，再按哈希分表（如 16/32 张 `product_xx` 分表）。
  3. 热点隔离：大卖家可独立库或独立分片。
  4. 历史归档：`InventorySyncLog` 按月归档，降低主库写放大。

## 5. 生产环境建议
- 中间件：可采用 [ShardingSphere](https://shardingsphere.apache.org/) 统一分片路由。
- Django 多数据库路由：
  - 将写请求路由至主库；
  - 读请求按规则路由至从库；
  - 平台维度可配独立 `DATABASES` + `db_router`。
- 大批量写入建议：
  - 采集阶段使用 `bulk_create/bulk_update` 替代逐条写入；
  - 对 `Product` 使用批次 UPSERT（MySQL `ON DUPLICATE KEY UPDATE`）优化吞吐。
