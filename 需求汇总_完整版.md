# 豪哥需求文档汇总


## 库存管理前后端.docx

库存管理模块：动态预警与多仓协同(Inventory)
为了让景琦和嘉瑞能够高效落地，这份详细方案将业务逻辑、技术实现细节和演示亮点进行了深度融合。你可以直接将对应的部分发给他们。
一、 订单管理模块：全链路状态机 (Order Management)
这个模块的核心目标是解决**“信息孤岛”**。跨境订单来自不同平台，嘉瑞需要将其标准化，景琦需要将其可视化。
1. 前端实现建议 (景琦)
状态卡片（Status Tiles）： 在页面顶部增加一组卡片，显示“待审核、待发货、物流异常、已完成”的实时数字。
平台标识系统： 在表格中，利用 Element Plus 的 el-tag 配合自定义 Icon（TikTok, Amazon, Shopee），让用户一眼看出订单来源。
高级筛选器（Filter Bar）： * 支持模糊搜索（订单号、SKU、买家）。
支持时间跨度选择（下单时间 vs 付款时间）。
订单操作抽屉 (Drawer)： * 右侧弹出，展示收货人地址、商品清单（含缩略图）、买家留言。
亮点： 增加“订单修改日志”，记录谁在什么时候修改了地址或备注。
2. 后端实现建议 (嘉瑞)
数据模型设计： * 建立 BaseOrder 抽象基类，包含通用字段。
利用 PlatformOrder 存储原始 JSON 报文，防止不同平台字段冲突导致数据丢失。
Celery 调度： * 设置定时任务（Periodic Tasks），每 10-30 分钟轮询一次 TikTok Shop API 的 order.list 接口。
处理 Rate Limit（限流），确保 API 密钥不会因请求过快被封禁。
权限控制： 只有具备 order_edit 权限的角色（RBAC）才能点击“手动修改地址”按钮。
二、库存管理模块：动态预警与多仓协同(Inventory)
库存是跨境企业的生命线。我们要展示的是“自动化”和“安全性”。
1.前端实现建议(竞祺)
库存预警雷达：利用el-table的row-class-name属性，当stock<min_stock时，整行高亮显示（如浅黄色预警，深红色断货）。
库存变动折线图：在SKU详情页，用ECharts展示该商品的库存消耗速率，预测还能卖几天。
多仓视图：通过Tab切换查看“深圳仓”、“海外仓（英国）”、“在途库存”
后端实现建议(嘉瑞)
嘉瑞可以在后端实现这个逻辑，当实际库存低于该值时，通过DjangoSignals发送系统通知。
原子性操作：在处理订单扣减库存时，必须使用db.transaction.atomic()，防止在高并发下出现“库存负数”。
SKU 映射系统：支持“多对一”映射。例如：TikTok上的SKU_A和 Amazon上的 SKU_A1实际上都对应仓库里的同一件商品。
注：这个公式的核心逻辑是“为极端情况准备的缓冲垫”。在跨境ERP中，最怕的就是由于供应商发货慢（物流延迟）或者某天突然爆单（销量激增）导致没货卖。这个公式就是为了算出：在最倒霉的情况下，我们需要额外存多少货才能不掉链子。可以去检索一下该逻辑。


## 物流追踪前后端.docx

1.前端实现建议(竞祺)
动态轨迹地图：集成地图SDK（如Mapbox或高德地图）。
根据订单的origin和destination绘制抛物线。
当包裹到达某一节点（如“清关中”）时，该节点在地图上变亮。

细节优化：如果物流超时，时间轴节点颜色变为红色。
2.后端实现建议(嘉瑞)
物流聚合层：不要直接对接几十个物流商。嘉瑞应接入17Track或快递的API。
统一物流轨迹格式：[{"time":"2026-04-16","status":"已揽收","location":"深圳"}]。
主动推送(Webhook)：配置Webhook接收物流平台的更新回调。一旦检测到“投递成功”，后端自动更新订单状态为“已签收”。
运费预估：根据包裹体积重和目的地，实时查询物流商价格表，给运营人员参考。

## 达人检索后端.docx

达人主干数据表 (核心档案表）
这是整个模块的基础，记录达人的静态和联系信息，支撑多平台数据聚合 。
平台与身份标识：需要存储达人在系统内的唯一 ID，以及对应社交平台的账号标识（Handle）。
基础检索维度：记录达人所属的国家/地区（如 US、UK、SEA 等）、主要使用语种。
业务分层标签：存储粉丝数等级字段，明确其属于 Nano、Micro 还是 Mega 级别 ，方便后续的高效筛选。
建联渠道信息：结构化存储达人的联系方式，包括公开的 Email 地址、WhatsApp 号码以及 Instagram DM 链接 ，支持前端在 ERP 内部直接发起联系 。
时区辅助字段：存储达人所在地的标准时区，后端需配合定时任务或动态计算，以支持前端展示达人所在地的当前时间并提示最佳建联时段 。
电商属性与受众画像表 (动态指标表)
该表建议与主表做一对一或一对多关联，用于存储高频变动的数据，支撑前端的达人看板和增长曲线图 。
受众画像数据：细化存储粉丝数据，包括年龄分布比例、性别比例以及预估的购买力等级 。
Amazon 平台关联：布尔值记录其是否拥有 Amazon Storefront，并存储对应的链接及在亚马逊上的带货排名数值 。
TikTok 平台关联：记录达人与 TikTok Shop 的关联状态，存储其历史带货类目列表，并记录核心的 GPM（千次播放成交额）数据
热度历史记录：需要设计一个时间序列表，定期（如每日/每周）快照达人的粉丝数和视频热度，以便前端绘制热力图和增长曲线 。
AI 内容分析与评价聚合表 (智能辅助表)
承接系统的 AI 处理结果，供业务端进行达人评估与背调 。
内容调性标签：存储 AI 分析达人过去 30 天视频后输出的风格分类结果 ，如“硬广推销型”、“测评拆解型”或“生活方式植入型” 。
竞品反查映射：记录达人推广过的竞品信息。需要一个关联字段对应输入的竞争对手品牌名或产品链接 。
受众意向分析：存储 AI 抓取并提取的评论区高频关键词（如“Where to buy”、“Too expensive”） ，以及系统计算出的粉丝真实购买意向评分 。
履约管理与授权素材表 (ERP 联动与闭环)
这部分直接关系到咱们内部后端的业务流转，特别是跨模块的协同。
样品订单流转：状态机字段，记录达人合作状态。当达人同意后，需触发逻辑将其转为“客户”，并生成关联的 0 元样品订单 ID 。
WMS 物流状态同步：关联存储跨境物流（如云途、4PX、递四方）的运单号和实时状态轨迹 ，并设置状态变更的回调逻辑，以便自动触发“快递已签收”的通知 。
TikTok 授权管理：安全存储达人提供的 TikTok Spark Ads 授权代码（Auth Code） ，并记录其有效期限字段 ，需配合过期预警机制。
AI 素材库回传归档：记录达人高清原片上传后的云端存储路径（URL） 。这一块需要与系统的“AI 素材库”打通，确保文件状态标记为“可提取”，供后续 AI 生成视频模块调用进行二次混剪 。
后端需提供给前端的特殊接口支持
除了常规的增删改查（CRUD），嘉瑞在后端还需额外封装以下功能接口以支撑业务：
AI 多语种邀约接口：接收前端传来的达人 ID（用于提取风格）和目标语种参数，后端调用大模型 API 生成个性化的邀约信（Collaboration Pitch）并返回文本结果 。
看板聚合查询接口：由于前端需要展示“达人看板”，后端需要提供一个聚合接口，一次性返回达人的基础信息、粉丝增长曲线时间序列数据以及热力图所需的热度指标 ，避免前端多次冗余请求。

## 选品决策算法前端.docx

模块名称：选品决策与 ROAS 测算沙盘 技术栈：Vue 3 + Element Plus + Pinia + Axios + Vite
1. 交互界面设计 (UI/UX)
测算工作台：
搜索引入组件：支持通过 SKU 或商品名称搜索，选中后自动从后端拉取该商品的基础成本（采购价、运费）。
动态参数输入区：使用 el-form 配合 el-input-number 或 el-slider，让用户手动调节“预期宣传费用”和“预估销量/GMV”。
可视化结果区：
核心指标看板：突出显示“预估收入”、“总投入”以及最重要的“ROAS 比值”。
决策灯号系统：
比值 < 3：el-tag 显示红色（高风险）。
3 ≤ 比值 ≤ 5：el-tag 显示绿色（健康/推荐）。
比值 > 5：el-tag 显示金色并带有闪烁或高亮动画（爆款潜力）。
2. 状态管理 (Pinia)
DecisionStore：
currentProduct：存储当前选中的商品基本属性。
calcParams：存储用户调节的即时参数（宣传费、预期 GMV 等）。
lastResult：存储最近一次接口返回的计算结果，用于对比展示。
3. 网络请求与联动 (Axios)
防抖计算：在用户调节参数时，使用防抖逻辑触发 API 请求，避免高频调用。
接口对接：
GET /api/v1/products/：获取选品列表。
POST /api/v1/decision/calculate/：发送参数，获取 ROAS 决策建议


## 选品决策算法后端.docx

模块名称：选品决策算法引擎 技术栈：Python + Django + DRF + Celery + RBAC
1. 数据库模型 (Models)
DecisionLog (决策日志表)：
记录每次测算的用户 ID（关联 User 模型）、商品 ID、输入的费用参数、计算出的比值。
作用：用于后续分析“业务员的选品审美偏好”以及“决策成功率”。
AlgorithmConfig (算法参数表)：
存储 3 和 5 这两个阈值。这样如果未来生意逻辑变了（例如提高盈利要求到 4~6），只需修改配置表而无需改动逻辑。
2. 核心逻辑与 API 设计 (DRF)
RBAC 权限控制：
仅具有“选品决策员”或“管理层”权限的角色可以访问测算模块。
计算视图 (CalculateView)：
逻辑流：接收 product_id -> 从数据库获取采购价、实时运费（联动 WMS）和平台佣金率 -> 结合前端传来的“宣传费用” -> 执行公式 预估收入 / (宣传费用 + 固定成本 + 变量成本) -> 返回 ROAS 数值及决策标签。
异常处理：处理除数为零（宣传费用为 0 时）的逻辑，并返回友好的错误码。
3. 异步任务与性能 (Celery)
批量测算任务：
当用户选择“商品 A”并一键测算“检索出的 100 个达人”时，同步请求会阻塞。
静良需启动 Celery 异步队列，逐个计算商品与达人的碰撞比值，计算完成后通过 WebSocket 或前端轮询更新列表中的“建议等级”。
